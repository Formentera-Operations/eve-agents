# Wizard Ideas — doc-intel / eve-agents (CC)

Method: read the full repo surface (agent tools/skills/instructions, analysts service, evidence + graph modules, all decisions, references, corpus/benchmark docs, eval suite, build-history memory), generated 30 candidate ideas internally, scored each on utility-to-complexity for Formentera specifically, and winnowed to the five below, ordered best to worst.

---

## 1. The Reconciliation Leg — documents vs. systems of record

**What it is.** A fourth retrieval leg that lets doc-intel answer questions no single system can: *"How did the Atlas J.J. 4H AFE estimate compare to actual GL spend?"*, *"Does the well test in the file match what the production DB shows for that month?"*, *"Which vendors appear in the Bull Mountain well files but have no GL activity on those wells?"* The three existing legs recall what the corpus means, says, and contains; this leg checks the corpus **against the business** — Snowflake masters, the AtScale semantic layer, EnergyLink JIB/revenue data.

**How it works technically.** eve already reserves the slot: `agent/connections/` ("external MCP/OpenAPI servers, one per file" — `references/eve-conventions.md`). Formentera already runs the servers: the AtScale MCP (`list_models` / `describe_model` / `run_query` over governed models) and the governance MCP. Drop a read-only `agent/connections/atscale.ts` into `agents/doc-intel/agent/`, add a `reconcile.md` runtime skill beside `agent/skills/citations.md` teaching when to cross-check and how to cite it, and extend the citation contract the way PR #29 already did for the ledger: system-of-record claims cite as **(model, measure, filter)** the way ledger claims cite as (S3 key, ledger status, as of `ledger_as_of`) — the precedent for non-page citation forms is already merged.

The decisive detail is that **the join key already exists and was built for exactly this**. `CONCEPTS.md` (Ontology Validity): "matching rewrites identity, so downstream consumers join validated entities to the ontology by name." The 490 banked named individuals in `references/ontology/welldrive.owl` are minted from `gold_dim_well` / `gold_dim_vendor` (`references/ontology/masters/*.sql`). So when `query_knowledge_graph` returns `liberty_energy` or a conformed well name, that string is a valid key into the warehouse. Rob has also already identified the authoritative fact table: GL (`GOLD_FCT_LOS`) beats invoice extracts for vendor-well linkage (noted 2026-07-09). Nothing about the ontology work needs to change; this idea cashes the check that work wrote.

**Why it's accretive for Formentera specifically.** AFE-vs-actuals variance is a core cost-control workflow at every operator, and today it requires a person to hold an AFE PDF in one hand and a Snowflake query in the other. doc-intel holds the AFE page-cited (the `financial_analyst` class in `references/analyst-classes.json` already owns AFE / Field Ticket / Workover entry types) and the warehouse holds actuals behind a governed semantic layer. No vendor product joins these two, because no vendor has both the page-cited archive and the conformed entity spine. This is the feature that turns doc-intel from "search my files" into "audit my business," and it is only buildable here.

**Implementation sketch.**
1. `agent/connections/atscale.ts` — read-only MCP connection, governed models only (the semantic layer is the sanctioned query surface; no raw LLM-SQL against Snowflake).
2. `agent/skills/reconcile.md` — routing: when a question compares a document figure to actuals, resolve entities to ontology-valid names first, query the semantic layer second, present both with dual citations; never average the two sources silently (mirrors the existing "if two documents disagree, present both" rule).
3. Instructions bullet defining the (model, measure, filter) citation form.
4. One eval, same shape as `evals/graph-pad-tandem.eval.ts`: an AFE-vs-actuals question that must call both `read_parsed_document` (or `delegate_analysis`) and the connection tool, and must carry both citation forms. Fail-closed gates per the PR #17/#23/#28 lesson.
5. Non-goals for v1: no writes, no EnergyLink actions, no new Python code — this is an eve-side leg.

**Confidence rationale.** High on feasibility: every component exists (connections slot, running MCP servers, conformed names, citation-form precedent, eval pattern). The risks are prompt-shaped, not architecture-shaped: entity resolution misses (mitigated — the matchable-ceiling work measured the matcher at 100% efficiency; misses are visible in the unverified report) and query hallucination (mitigated — AtScale's `describe_model` grounds the schema, and the semantic layer rejects invalid queries loudly). The main open question is which two or three governed measures to bless for v1 (AFE gross estimate vs. cumulative DEV CAPEX by well is the obvious first pair). Scoped to one connection + one skill + one eval, it honors the simplicity-first rule while changing what the product *is*.

## 2. Ship the Legs to the Claude Estate — doc-intel as an MCP server

**What it is.** Expose doc-intel's retrieval legs as a read-only MCP server so the company's existing claude.ai / Claude Code surfaces can call them today — before the Vercel production deploy, Teams channels, or the Phase 3 billing migration land. Formentera already lives in an MCP estate (EnergyLink, AtScale, PatchOps, M365, and notably `welldrive-files`, a *raw* archive browser). doc-intel-as-MCP is the intelligence upgrade to that last one: semantic + exact + visual search with page citations and ledger-grounded coverage answers, instead of list-and-read.

**How it works technically.** The analysts service (`analysts/src/doc_intel_analysts/service.py`) already exposes exactly the right HTTP endpoints: `/evidence/search`, `/evidence/grep`, `/evidence/find`, `/evidence/status`, `/evidence/read` (including service-side vision — screenshots never leave the service, per `decisions/2026-07-06-evidence-read-vision-mechanism.md`), and `/graph/search`. An MCP layer is a thin adapter: mount a FastMCP (or `mcp` SDK) app on the same FastAPI process, one MCP tool per endpoint. The critical asset is free: **the eve tool descriptions are the prompt engineering**. `check_document_status.ts`, `search_evidence.ts`, `grep_evidence.ts` etc. carry the hard-won coverage semantics (sample vs. tranche scope, contiguous-substring warnings, `ledger_as_of` dating, "never quote manifest counts as corpus size") in their `description` fields — port them verbatim as the MCP tool descriptions and any competent client model inherits the discipline that took PRs #15–#29 to encode. Add a `usage` prompt/resource distilled from `agent/instructions.md` (citation contract, absence-claim contract) for clients that read MCP resources.

**Why it's accretive for Formentera specifically.** Distribution is currently the project's weakest axis: 6 evals, a 25-question benchmark, ledger-grounded negatives — and one user, on one laptop. Every other capability in this document multiplies by the number of people who can reach the tools. This path reaches them without waiting on the two known blockers (gateway billing on Rob's personal scope; Phase 2 hosting), because it rides the same claude.ai connector pattern the company already administers, and because the marginal client (an engineer asking "which pages discuss stuck pipe on Westlake wells?" in claude.ai) needs no new habits. It also sets up idea 1's inverse: the same Claude session can hold doc-intel *and* AtScale *and* EnergyLink connectors, so cross-system workflows emerge in the client even before the eve-side leg ships.

**Implementation sketch.**
1. `analysts/src/doc_intel_analysts/mcp.py` — MCP server mounted at `/mcp` on the existing service; tools: `search_evidence`, `grep_evidence`, `find_evidence_files`, `read_evidence`, `check_document_status`, `query_knowledge_graph`. Read-only by construction (those endpoints have no write path).
2. Descriptions copied from the eve tools; a `coverage` MCP resource carrying the per-leg coverage model.
3. Auth: bearer token env var; the service currently binds localhost — expose via the Phase 2 host when it lands (the MCP layer is host-agnostic, so this work is not wasted by the hosting decision).
4. Explicitly *not* replacing the eve agent: eve remains the opinionated front door that enforces verification (the seam decision's "the agent that owns the user relationship never repeats an uncited claim"); the MCP surface is the raw-legs tier for sophisticated clients, exactly as `welldrive-files` is today for raw files.
5. Later, when Phase 3 unblocks production deploy, the eve agent itself gets channels (Teams) — this MCP layer is the bridge, not the terminus.

**Confidence rationale.** Very high on feasibility — it's an adapter over tested endpoints, and the org demonstrably knows how to run MCP connectors at scale. The honest risk is governance: an MCP client model is *not* bound by `instructions.md`, so citation discipline degrades to "as good as the tool descriptions plus the client model." I rate that acceptable for a read-only surface whose every hit carries `s3key` + `page_num` provenance by schema (the retrieval layer, not the prompt layer, enforces provenance — that was the architecture's smartest bet, and this idea is where it pays off). Smallest lift of the five; highest adoption leverage.

## 3. The Data-Room Product — per-well completeness audit over the ledger

**What it is.** A batch product that answers, per well: *which document classes should exist, which do, which are present-but-unreadable, and which are missing* — rendered as a per-well / per-pad gap report. Two buyers inside the company: **sell-side** (divestiture data-room readiness — Formentera's PE-backed model makes asset packages a recurring event, and assembling a defensible well-file data room is exactly this checklist) and **compliance/ops** (regulatory filings like W15s, cement reports, deviation surveys that must exist for every drilled well).

**How it works technically.** This productizes the machinery that already distinguishes doc-intel from generic RAG: the **ingest ledger** makes absence claims grounded (complete / deliberately-skipped-with-reason / failed, per `CONCEPTS.md` and `EvidenceRetriever.document_status`), and the manifest + `find_evidence_files` enumerate what exists per well. The expected-document matrix does not need inventing: `references/ontology/welldrive.owl` already encodes a DocumentClass hierarchy — one leaf per WellDrive entry_type (59), grouped per analyst class. Add a small open knowledge-layer table (`references/expected-documents.json`, same pattern as `analyst-classes.json` — data, never code branches) mapping well lifecycle stage → required entry_types (a drilled+completed producer needs AFE, Final Deviation Survey, Casing Tally, Cement Report, Frac Report, W15, …). A batch runner in the analysts service walks wells (from the manifest's `well` column and evidence-store `documents` table), joins expected vs. found vs. ledger status, and emits a report: green (indexed, page-cited), yellow (exists but deferred-format or unreadable — named, with the skip reason, dated by `ledger_as_of`), red (no candidate found, scoped honestly per the coverage rules: a genuine negative for Westlake, a sample-scoped caveat elsewhere).

**Why it's accretive for Formentera specifically.** It converts doc-intel's most unusual engineering investment — grounded negatives — into a business artifact executives can hold. Nobody else's document-AI can say "the cement report for this well exists but is a deferred-format spreadsheet, as of the July 10 ingest pass" and be *right*; that sentence is this system's signature, and a gap report is that sentence at portfolio scale. Buy-side, the same machinery inverts: `evidence/ingest.py --prefix` is already corpus-agnostic, so an acquisition target's data-room export dropped in S3 gets the identical treatment during diligence — coverage report first, questions second. That reframes doc-intel from "our archive's agent" to "an archive-diligence capability."

**Implementation sketch.**
1. `references/expected-documents.json` — lifecycle-stage → required entry_types table (Rob curates; it's domain judgment, kept as open data).
2. `analysts/src/doc_intel_analysts/audit.py` — deterministic joiner (no LLM in v1): manifest + `documents` table + ledger → per-well rows; CLI like the ingest module; output CSV + Markdown to `s3://formentera-welldrive-derived/runs/doc-intel/audit/<stamp>/`.
3. An eve tool `audit_well_file(well)` for interactive single-well use, same seam shape as `check_document_status.ts`.
4. Westlake first (the only team with full-tranche truth); other teams report at sample scope with the standard caveat until their tranches ingest.
5. Later: eve `schedules/` re-runs it on ledger change and diffs — but v1 is a CLI, not a scheduler.

**Confidence rationale.** High. The joins are mechanical over data that already exists and is already tested (the ledger got a dedicated tool, eval, and parity test in PR #29). The one genuinely new artifact — the expected-documents matrix — is a domain-knowledge table, cheap to draft and iterate, and honest about being judgment (kept in the open knowledge layer, versioned). Risk: well identity is stringly (directory names carry legal-location tokens); mitigated because the audit groups by the manifest/store's own `well` field rather than reconstructing names, and the ontology individuals offer a conformance path when it matters. Deterministic v1 means no eval burden and no gateway spend.

## 4. Un-park the Spreadsheets — close the deliberate coverage hole

**What it is.** Promote the Excel family out of "ledgered skip" into the evidence store. Deferred formats (Excel, XML, EML, ZIP — the v1 scope boundary in `references/evidence-store.md`) are the single largest class of *present-but-unsearchable* documents, and they are where engineering-grade numbers concentrate: frac per-stage workbooks, directional exports, AFE workbooks, cost trackers. Today the agent's best answer for these is the (excellent, honest) "exists, but it is a deferred-format spreadsheet." This idea makes the next answer "here is the value, cited."

**How it works technically.** Follow the text-native gate's existing pattern (`evidence/parse.py`): a workbook becomes one logical page **per sheet** (sheet name in the page record), serialized with openpyxl to pipe-delimited text, capped like the 2M-char text-native rule; the R10 Option B economics carry over unchanged (head-50-chunks embed; grep and page reads see full text) — so the cost is embeddings only, and the pilot measured embeddings at $2.43 for 24.4M chars, i.e. this is single-digit dollars at Westlake scale. Two pieces of existing machinery matter:

- **Terminality is checksum-scoped by design** (`CONCEPTS.md`: "skips are terminal for the exact bytes they judged"), and gate verdicts are policy, not history. Add a `gate_version` to the ledger row (or a one-time targeted reopen of `status='skipped', reason like 'deferred%excel%'`), mirroring how PR #12 re-opened the nine miscounted parse failures. The ledger's resumable-pass design means the reopened set re-ingests incrementally under `--max-new` batches with zero risk to settled rows.
- **The eval suite already anticipates this**: `evals/ledger-status.eval.ts` uses a dynamic fixture that "fails loudly at setup only if the corpus holds no skips at all — retarget this eval" — the roadmap for this exact un-parking is written into the test.

**Why it's accretive for Formentera specifically.** The wildcat enrichment alone set aside 1,308 frac CSVs and 241 xlsx from one DSU; per-stage frac data is what completions engineers actually ask about, and stage-level treating-pressure/proppant questions currently dead-end at the skip verdict. It also strengthens every other idea here: the completeness audit's yellow rows turn green, absence claims get stronger (fewer carve-outs to recite), and the reconciliation leg gains the cost workbooks that sit between the AFE PDF and the GL. Coverage compounding, not feature addition.

**Implementation sketch.**
1. `evidence/parse.py`: `_parse_excel` (openpyxl, values-only, per-sheet logical pages; formulas rendered as cached values); route `.xlsx/.xlsm/.xls` in the format gate; leave XML/EML/ZIP deferred (each is its own decision — don't bundle).
2. Ledger: `gate_version` column or scripted reopen; instructions + `check_document_status.ts` description updated to drop the "spreadsheets" example from the deferred list (the eval's dynamic fixture self-heals).
3. Run as a Westlake-prefix incremental pass — post-Phase 4 hosting ideally, but the laptop can do it under `--max-new` discipline (the crash-learned rule is already in memory: manifest passes always carry `--max-new`).
4. Success gate: 3–5 new benchmark questions drawn from spreadsheet-only facts (the benchmark README's process reproduces), pass alongside the existing 25.

**Confidence rationale.** High on value and mechanism, medium on schedule — the honest dependency is where the pass runs (the laptop's memory ceiling is well documented; Phase 4's Container Apps Job is the right home). No new dependencies if openpyxl is acceptable (it's the boring choice; a dependency addition needs Rob's explicit approval per house rules, which I'd flag rather than assume). Sheet-as-page is a judgment call — giant sheets will make text-dense pages — but it exactly mirrors the accepted LAS/CSV trade (full grep, capped semantic granularity), so it introduces no new epistemics.

## 5. Well Briefs — the answer ledger's pragmatic v1, shipped as a product

**What it is.** A per-well, page-cited, continuously-refreshable **brief**: spud-to-TIL narrative, key figures (AFE estimate, TD, stages, test rates), vendor roster, notable NPT events — every line carrying its (S3 key, page) citation, persisted as first-class data with the `ledger_as_of` watermark it was built against. This is the deferred answer-ledger decision (`decisions/2026-07-07-context-layer-answer-ledger.md`) entered through the front door it left open: push-side, from the verified-answer path, with no Agent Runs dependency and no cognee upgrade.

**How it works technically.** The deferral had two build-twice worries (Agent Runs traces as the natural feed; memory stage 2 overlap) and three cognee-adoption constraints. Well briefs sidestep all of them: a batch runner drives the **existing** verified-answer machinery — fixed question templates per well through `delegate_analysis`'s seam (`POST /analyze` with the well's parseable document set) or through a headless eve session (the documented `POST /eve/v1/session` pattern), then re-verifies citations exactly as `instructions.md` mandates — and persists only what survives verification. Storage is deliberately boring: Markdown + a JSON sidecar (question, answer, citations, ontology-valid entities, `ledger_as_of`, model, timestamp) under `s3://formentera-welldrive-derived/runs/doc-intel/briefs/<well>/`, following the derived-bucket doctrine ("everything computed *from* the corpus… can be rebuilt or discarded without touching sources"). A `read_well_brief` eve tool serves it back instantly; staleness is a ledger diff — new completed docs for a well since the brief's watermark mark it stale, and the audit table from idea 3 tells you which wells to refresh.

**Why it's accretive for Formentera specifically.** Three compounding payoffs. First, latency and cost: entity-shaped "tell me about the Bull Mountain pad" runs (the tandem evals) burn a multi-tool live loop every time; a brief is that answer, verified once, served forever, refreshed only on change — the compounding institutional memory the answer-ledger decision named as the prize. Second, it is the demo artifact: a one-page cited well card is what a drilling engineer, a reservoir engineer, and a managing partner can all evaluate in thirty seconds — far more legible than a chat transcript. Third, it is the answer ledger's schema learning lab: after fifty briefs you know empirically what verified-answer persistence needs (entity keys, staleness semantics, citation density) before committing to the fourth-layer architecture — building it once, informed, instead of twice.

**Implementation sketch.**
1. `references/brief-questions.json` — the per-well question template set (open knowledge table, same pattern as analyst classes).
2. `analysts/src/doc_intel_analysts/briefs.py` — batch runner: resolve a well's document set (manifest + evidence `documents` table), call the seam, verify citations against the store (`get_page` — mechanical containment check), write brief + sidecar; refuse to persist any uncited line (the "fact that cannot be pinned to a page is dropped" rule, enforced in code).
3. `agent/tools/read_well_brief.ts` + an instructions bullet: serve the brief when fresh, disclose the watermark, fall back to live legs when stale or absent.
4. Pilot scope: the ~10 Bull Mountain + Wildcat Hollow wells the graph already knows deeply — the briefs double as a human-readable audit of the $94 enrichment investment.
5. Cost gate before scaling (a Human Gate, per `CONCEPTS.md`): measure $/brief on the pilot set; extrapolate before running Westlake-wide.

**Confidence rationale.** Medium-high. Every mechanism is proven (the seam, citation verification, derived-bucket publishing, ledger watermarks); the genuinely new code is a batch driver and one read tool. The risk is product, not technical: briefs are only as good as the question templates, and a stale-but-confident brief is a new failure mode — which is why the watermark travels *inside* the artifact and the read tool discloses it (the same epistemic honesty the absence-claim contract already enforces). Ranked fifth not because it's weakest in ceiling but because it deliberately spends gateway dollars and its value multiplies with ideas 2 and 3 in place — sequence it after distribution and the audit table exist.

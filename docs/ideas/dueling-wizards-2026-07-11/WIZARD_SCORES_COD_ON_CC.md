## 1. The Reconciliation Leg — 790/1000

The product thesis is excellent: combining page-cited operational documents with governed financial and production facts would move doc-intel from archive search into genuinely differentiated audit work. The repository confirms both the current three-leg boundary ([instructions.md](/Users/robstover/Development/formentera/eve-agents/agents/doc-intel/agent/instructions.md:33)) and eve’s intended MCP connection slot ([eve-conventions.md](/Users/robstover/Development/formentera/eve-agents/references/eve-conventions.md:38)). AtScale’s current MCP also provides the claimed discovery and SELECT-only query tools, so that integration mechanism is credible. The dual-source citation principle is exactly right.

The implementation is substantially understated, however. The 490 ontology individuals are real, but the generator deliberately keeps the Snowflake master key out of the committed OWL and preserves corpus-side spellings as graph identities ([individuals.py](/Users/robstover/Development/formentera/eve-agents/agents/doc-intel/analysts/src/doc_intel_analysts/graph/individuals.py:158)). Those names join to the ontology, not reliably to `EID`, `VENDOR_ID`, or a semantic-model key. The enrichment result was 293/337 ontology-valid entities; “100% matcher efficiency” only means no runtime misses against individuals already available, not complete entity resolution ([selective-graph-enrichment decision](/Users/robstover/Development/formentera/eve-agents/decisions/2026-07-10-selective-graph-enrichment.md:24)). The repository contains no AtScale connection, `GOLD_FCT_LOS` contract, or proof of the claimed running corporate servers. A defensible implementation therefore needs an explicit ID bridge, blessed grain/date/currency rules, query IDs and as-of timestamps, and numerical eval fixtures—not merely one connection, skill, and “called both tools” assertion. `(model, measure, filter)` is too weak for a financial audit trail.

**Posture comparison:** the batch AFE mart is better first. It can normalize AFE versions, gross versus net amounts, stable well IDs, spend cutoffs, and page citations once, then join deterministically to tested GL facts. The live AtScale leg is valuable afterward as the conversational exploration layer over that mart and a small whitelist of governed measures. As proposed, this deserves a constrained trial, not direct production adoption.

## 2. Ship the Legs as an MCP Server — 620/1000

The adapter itself is feasible. The service already exposes the six relevant evidence operations plus graph search ([evidence/api.py](/Users/robstover/Development/formentera/eve-agents/agents/doc-intel/analysts/src/doc_intel_analysts/evidence/api.py:92)), and the official Python MCP SDK supports mounting a Streamable HTTP application beside an existing ASGI service. Reusing the carefully refined coverage descriptions would improve arbitrary clients, and MCP is genuinely useful for engineer-facing agent composition with AtScale and EnergyLink.

It does not, however, solve distribution “before hosting.” Retrieval still opens a fixed local LanceDB path ([config.py](/Users/robstover/Development/formentera/eve-agents/agents/doc-intel/analysts/src/doc_intel_analysts/evidence/config.py:24)), while Claude remote connectors must be publicly reachable from Anthropic’s infrastructure and normally need an enterprise authentication flow ([Claude connector requirements](https://support.claude.com/en/articles/11175166-get-started-with-custom-connectors-using-remote-mcp)). Thus Phase 2 hosting remains a hard dependency. The Python package has no MCP dependency or production auth middleware, and exposing only `/mcp` behind a shared bearer token would leave the existing REST routes to secure separately. More importantly, provenance fields do not enforce epistemic discipline: search hits contain pages, but clients can cite snippets without reading them; graph results remain document-level; and nothing forces ledger-aware absence claims. This bypasses the accepted rule that eve owns the truth and verifies claims before presenting them ([seam decision](/Users/robstover/Development/formentera/eve-agents/decisions/2026-07-05-doc-intel-seam.md:24)).

**Posture comparison:** the Teams desk is the better first distribution surface. It requires the same public hosting work but retains eve’s instructions, verification loop, sessions, and company-facing identity model while meeting humans where they already work. MCP should follow for sophisticated AI clients after OAuth, rate limiting, pagination/output caps, read-only annotations, and client-independent coverage resources are implemented. It is a useful second channel, not the claimed shortcut around deployment.

## 3. The Data-Room Product — 705/1000

This is a strong product direction. It turns the ledger’s unusual ability to distinguish indexed, deliberately skipped, and failed documents into an artifact useful for divestiture preparation and archive diligence. Westlake is the correct first scope, and a deterministic CSV/Markdown output is preferable to an LLM-generated completeness opinion. The ontology does contain the claimed 59-entry-type hierarchy ([ontology README](/Users/robstover/Development/formentera/eve-agents/references/ontology/README.md:6)), while the ledger supports honestly dated Westlake negatives.

The proposed join is not currently mechanical. The full-tranche `documents` table stores `doc_id`, key, team, format, and page count—but neither `well` nor authoritative `entry_type` ([store.py](/Users/robstover/Development/formentera/eve-agents/agents/doc-intel/analysts/src/doc_intel_analysts/evidence/store.py:48)). Prefix ingest likewise records only key, team, and ETag ([ingest.py](/Users/robstover/Development/formentera/eve-agents/agents/doc-intel/analysts/src/doc_intel_analysts/evidence/ingest.py:39)). A well can be recovered from the path, but entry type must come from S3 metadata; repository policy explicitly forbids inferring it from filenames. Lifecycle stage is also absent. Finally, the ontology enumerates document classes but says nothing about requiredness. A versioned expectation matrix is substantial operating-policy work involving well type, jurisdiction, vintage, former operator, and lifecycle—not a cheap final table. Incorrect red gaps would be dangerous in a compliance or transaction report.

**Posture comparison:** the readiness radar is better at the current maturity because it can rank observed coverage, failures, deferred formats, freshness, and evidence density without asserting that a document legally “should” exist. The deterministic audit becomes the stronger end-state after enriching the full-tranche catalog with well IDs and entry types and obtaining owner-approved expectation rules. A Westlake pilot for one vetted well archetype would be justified.

## 4. Un-park the Spreadsheets — 750/1000

The coverage diagnosis is sound. The approved evidence-store plan records 4,661 Excel-family files—most of the roughly 5,100 deliberate format skips—so this is the largest known hole ([evidence-store plan](/Users/robstover/Development/formentera/eve-agents/docs/plans/2026-07-06-002-feat-evidence-store-plan.md:107)). The proposal also correctly understands checksum terminality, the need to reopen prior skips, and the dynamic ledger eval. Expanding retrieval coverage is foundational and would benefit reconciliation, completeness reporting, and technical questions simultaneously.

The parser design needs rework. `openpyxl` is not a current dependency, does not read legacy `.xls` or `.xlsb`, and `data_only` returns only the value cached when Excel last calculated the workbook ([openpyxl reader documentation](https://openpyxl.readthedocs.io/en/stable/_modules/openpyxl/reader/excel.html?highlight=load_workbook)); the existing gate covers `.xls`, `.xlsx`, `.xlsm`, and `.xlsb` together ([parse.py](/Users/robstover/Development/formentera/eve-agents/agents/doc-intel/analysts/src/doc_intel_analysts/evidence/parse.py:147)). More importantly, “sheet as page” creates synthetic page citations that do not correspond to anything a human can open. Spreadsheet evidence needs a first-class `(S3 key, sheet, cell/range)` contract, with cell coordinates preserved in serialized text. Pipe-delimited values alone lose formula, merged-cell, layout, and provenance information. The single-digit embedding estimate is also unsupported without measuring workbook text volume and sparsity; the cited Wildcat-specific counts are not present in the committed repository.

**Posture comparison:** the demand-weighted flywheel is better for sequencing because actual blocked questions should determine which workbook cohorts and entry types receive parser investment. It will probably nominate Excel first, making this idea a likely implementation inside that flywheel. The right initial slice is `.xlsx`/`.xlsm` for a few demanded wells, preserving sheet/cell/formula/cached-value provenance and adding spreadsheet-only benchmark questions before an Azure-hosted bulk pass.

## 5. Well Briefs — 545/1000

The artifact is appealing. A concise, cited well card would be easier for engineers and executives to consume than a chat transcript, and Markdown plus structured JSON in the derived bucket is appropriately portable. Precomputing frequently requested well narratives could reduce latency and provide a useful demo once the underlying claim persistence exists.

The proposal’s central feasibility claim is false today: there is no programmatic “verified-answer machinery” to drive in batch. `delegate_analysis` accepts at most 25 documents and resolves them only through the 500-file sample manifest ([delegate_analysis.ts](/Users/robstover/Development/formentera/eve-agents/agents/doc-intel/agent/tools/delegate_analysis.ts:22)); full-tranche evidence documents cannot be handed through that seam. The service parses model-supplied citation JSON but does not verify it ([service.py](/Users/robstover/Development/formentera/eve-agents/agents/doc-intel/analysts/src/doc_intel_analysts/service.py:114)), while `get_page` proves only that a cited page exists—not that each persisted sentence is entailed by it. “Refuse any uncited line” would require claim-level structured output, citation-to-claim mapping, and semantic or human verification. A global `ledger_as_of` watermark is also insufficient for staleness; each brief needs the exact contributing document IDs, checksums, template version, and model version. These are precisely the missing persistence semantics behind the decision to defer the answer ledger ([answer-ledger decision](/Users/robstover/Development/formentera/eve-agents/decisions/2026-07-07-context-layer-answer-ledger.md:42)).

**Posture comparison:** the NPT loop is better. It targets a narrower, recurring operational workflow with clearer human feedback, and the current drilling analyst, graph enrichment, and evidence benchmark already recognize NPT events. Generic well briefs risk becoming expensive, stale summaries that duplicate existing well cards. Briefs should eventually be materialized views over a verified claim ledger and readiness catalog—not a shortcut used to create those foundations.
